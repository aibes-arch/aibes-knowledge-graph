# 01 需求分析

## 1.1 产品定位

**AIBES Domain Graph Builder** 是一款面向任意行业的**知识图谱自动构建与 GraphRAG 应用开发平台**。

其核心定位是：

> 让非结构化数据（文档、手册、记录、笔记等）自动转化为可查询、可推理、可可视化的领域知识图谱，并基于图谱实现增强检索与智能问答。

对标产品包括：
- Microsoft GraphRAG：从文本抽取知识图谱并生成社区摘要
- Neo4j LLM Knowledge Graph Builder：从非结构化内容抽取节点、关系、属性

## 1.2 目标用户

| 用户角色 | 核心诉求 |
|----------|----------|
| 领域专家 | 将经验、手册、标准转化为结构化知识 |
| 维修工程师 | 快速定位故障原因与维修方案 |
| 科研人员 | 从文献中抽取研究实体与关系 |
| 企业知识管理员 | 统一管理和检索企业知识资产 |
| AI 应用开发者 | 基于图谱构建 RAG / Agent 应用 |

## 1.3 适用领域

- 装备维修与故障诊断
- 智能钻井
- 法律案例与法条知识
- 医疗诊疗知识
- 制造业工艺知识
- 科研文献知识网络

## 1.4 核心功能需求（MVP）

根据第一版规划，MVP 仅实现 5 个核心功能：

1. 上传 PDF / Word / Markdown / TXT 文档
2. 自动解析文本并切块
3. 使用 LLM 自动抽取实体和关系
4. 将抽取结果写入 Neo4j 图数据库
5. 提供图谱可视化与知识问答界面

## 1.5 非功能需求

| 类别 | 要求 |
|------|------|
| 可扩展性 | 支持多领域 Schema 配置，不写死实体类型 |
| 可维护性 | 模块化设计，便于后续替换 LLM、存储、解析器 |
| 可部署性 | 提供 Docker Compose 一键部署 |
| 易用性 | 提供可视化界面，降低非技术人员使用门槛 |
| 开放性 | 提供 RESTful API，便于第三方集成 |

## 1.6 用户场景示例

### 场景 1：装备维修

> 维修工程师上传《风机冷却系统维修手册》，系统自动抽取装备、部件、故障、维修方案、工具、备件等实体及关系，生成知识图谱。工程师提问"风机异响如何处理"，系统基于图谱给出维修步骤与所需备件。

### 场景 2：科研文献

> 研究人员上传多篇论文 PDF，系统抽取论文中的方法、数据集、实验结果、引用关系，构建研究知识网络，辅助文献综述。

### 场景 3：法律案例

> 律师上传判决书，系统抽取案件、当事人、法条、判决结果等实体，构建案例知识图谱，支持类案检索与法条推荐。
