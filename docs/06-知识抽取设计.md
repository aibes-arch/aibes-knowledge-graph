# 06 知识抽取设计

## 6.1 抽取流程

```text
文本块
  │
  ▼
构建 Prompt（Schema 约束）
  │
  ▼
调用 LLM / Mock 抽取器
  │
  ▼
解析 JSON → Entity + Relation 列表
  │
  ▼
存入 Candidate（待审核/直接写入）
```

## 6.2 Schema 约束

抽取时必须遵守当前领域 Schema 中定义的实体类型和关系类型。

默认装备维修 Schema 约束示例：

```text
只允许使用以下实体类型：
Equipment, Subsystem, Component, Fault, Symptom, Cause, Solution, Procedure, Tool, SparePart, Supplier, Document, Person, Record

只允许使用以下关系类型：
HAS_SUBSYSTEM, HAS_COMPONENT, HAS_FAULT, HAS_SYMPTOM, CAUSED_BY, SOLVED_BY, HAS_STEP, REQUIRES_TOOL, REQUIRES_PART, SUPPLIED_BY, MENTIONS, CONFIRMS, PROVIDES
```

## 6.3 Prompt 模板

```text
你是知识图谱抽取专家。请从文本中抽取实体、关系和属性。

只允许使用以下实体类型：
{entity_labels}

只允许使用以下关系类型：
{relation_types}

输出严格 JSON，格式如下：
{
  "entities": [
    {
      "name": "实体名称",
      "type": "实体类型",
      "properties": {},
      "evidence": "原文片段",
      "confidence": 0.95
    }
  ],
  "relations": [
    {
      "source": "源实体名称",
      "target": "目标实体名称",
      "type": "关系类型",
      "properties": {},
      "evidence": "原文片段",
      "confidence": 0.9
    }
  ]
}

如果文本中没有明显实体或关系，返回空数组。不要编造。

文本：
{chunk_text}
```

## 6.4 输出结构

```python
class Entity(BaseModel):
    name: str
    type: str
    properties: Dict[str, Any] = {}
    evidence: str
    confidence: float

class Relation(BaseModel):
    source: str
    target: str
    type: str
    properties: Dict[str, Any] = {}
    evidence: str
    confidence: float

class ExtractionResult(BaseModel):
    entities: List[Entity]
    relations: List[Relation]
```

## 6.5 LLM 调用

使用 OpenAI-compatible 接口：

```python
client = OpenAI(base_url=..., api_key=...)
client.chat.completions.create(
    model=settings.llm_model,
    messages=[...],
    response_format={"type": "json_object"},
    temperature=0.0,
)
```

支持通过环境变量切换任意 OpenAI-compatible 服务：

```env
LLM_BASE_URL=https://api.openai.com/v1
LLM_API_KEY=sk-xxx
LLM_MODEL=gpt-4o-mini
```

## 6.6 Mock 抽取模式

为便于无 LLM Key 时演示，内置规则化抽取器。设置环境变量即可开启：

```env
MOCK_LLM=true
```

Mock 模式基于预定义关键词列表和共现句子生成实体与关系，适合快速验证端到端流程。

## 6.7 人工审核预留

候选数据默认状态为 `pending`，后续可扩展审核中心：

- `pending`：待审核
- `approved`：审核通过，可写入图谱
- `rejected`：审核拒绝
- `error`：抽取异常

## 6.8 实体对齐预留

当前 MVP 未做实体对齐，后续应实现：

1. 规则归一化：大小写、空格、符号统一
2. 同义词词典匹配
3. Embedding 相似度去重（> 0.88）
4. LLM 判断是否为同一实体
5. 低置信度进入人工审核队列
